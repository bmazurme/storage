# 📦 Складской учёт

Приложение для учёта остатков на складе с иерархией хранения:

**Стеллаж → Полка → Контейнер → Бокс → Юнит (позиция)**

## Возможности

- CRUD для всех уровней хранения с указанием **вместимости в юнитах** и индикацией заполненности (прогресс-бар: зелёный / жёлтый ≥80% / красный при переполнении).
- Юниты: название, описание, изображение (MinIO), количество, минимальный остаток.
- **Поиск** по названию, описанию и коду (регистронезависимый, ILIKE).
- **Индикация остатков**: 🟢 «Достаточно» · 🟡 «Заканчивается» (количество ≤ мин. остатка) · 🔴 «Нет на складе» (количество = 0).
- **QR-коды** для любого элемента хранения и юнита (генерация PNG на бэкенде, скачивание).
- Светлая и тёмная темы (Gravity UI, переключатель в шапке, сохраняется в localStorage).
- Быстрое изменение количества кнопками «+ / −» прямо в списке.
- **Drag-and-drop** сортировка юнитов внутри бокса (react-dnd) с сохранением порядка на бэкенде.

## Система кодировки

Код присваивается автоматически при создании и однозначно указывает физическое
местоположение. Формат: префикс уровня + порядковый номер внутри родителя.

| Уровень   | Префикс | Пример кода           |
|-----------|---------|-----------------------|
| Стеллаж   | `R`     | `R01`                 |
| Полка     | `S`     | `R01-S02`             |
| Контейнер | `C`     | `R01-S02-C03`         |
| Бокс      | `B`     | `R01-S02-C03-B04`     |
| Юнит      | `U`     | `R01-S02-C03-B04-U005`|

QR-код содержит эту строку — отсканировав его, сразу видно адрес хранения:
стеллаж 1, полка 2, контейнер 3, бокс 4, позиция 5.

## Стек

- **Backend**: NestJS, TypeORM, PostgreSQL, MinIO (изображения), `qrcode` (QR-генерация)
- **Frontend**: React 18, Vite, Redux Toolkit (RTK Query), react-hook-form, Gravity UI
- **Инфраструктура**: Docker, docker-compose, nginx

## Запуск через Docker

```bash
docker compose up -d --build
```

| Сервис          | Адрес                  |
|-----------------|------------------------|
| Приложение      | http://localhost:8080  |
| API             | http://localhost:3000/api |
| Консоль MinIO   | http://localhost:9001 (minioadmin / minioadmin) |

## Локальная разработка

Нужны запущенные PostgreSQL и MinIO (можно из compose: `docker compose up -d postgres minio`).

```bash
# backend (http://localhost:3000)
cd backend && npm install && npm run start:dev

# frontend (http://localhost:5173, проксирует /api на :3000)
cd frontend && npm install && npm run dev
```

Настройки backend берутся из переменных окружения
(`DB_HOST`, `DB_PORT`, `DB_USER`, `DB_PASSWORD`, `DB_NAME`,
`MINIO_ENDPOINT`, `MINIO_PORT`, `MINIO_ACCESS_KEY`, `MINIO_SECRET_KEY`, `MINIO_BUCKET`);
значения по умолчанию рассчитаны на локальный запуск.

> Схема БД создаётся автоматически (`synchronize: true`) — удобно для разработки;
> для продакшена стоит перейти на миграции TypeORM.

## API

| Метод | Путь | Описание |
|-------|------|----------|
| `GET` | `/api/racks/tree` | Полное дерево склада с юнитами |
| `POST` | `/api/racks` \| `/api/shelves` \| `/api/containers` \| `/api/boxes` | Создание (`name`, `capacity`, + id родителя) |
| `PATCH` / `DELETE` | `/api/{racks\|shelves\|containers\|boxes}/:id` | Изменение / удаление (каскадно) |
| `GET` | `/api/units/search?q=...` | Поиск юнитов по названию, описанию, коду |
| `POST` / `PATCH` / `DELETE` | `/api/units[/:id]` | CRUD юнитов |
| `PATCH` | `/api/units/reorder` | Сохранение порядка юнитов внутри бокса (`boxId`, `unitIds`) |
| `POST` | `/api/files/upload` | Загрузка изображения (multipart, поле `file`) в MinIO |
| `GET` | `/api/files/:key` | Получение изображения |
| `GET` | `/api/qr/:code` | PNG с QR-кодом |
